# Facts

- End-to-end acceptance: chunk payloads generated by the extension from a real Confluence page, pasted into the ADFReader scanner (simulator), reassemble and render the document correctly.
- Tools/adf-beam-extension loads as an unpacked MV3 extension in Chrome Developer mode with no errors.
- Clicking the toolbar action on a Confluence Cloud page fetches that page's ADF via /wiki/api/v2/pages/{id}?body-format=atlas_doc_format using the browser's existing session (no auth flow).
- The overlay shows a large cycling QR, the page title, a frame i/n indicator, and a close button.
- Each QR frame payload is 'ADF1|<docId>|<index>|<total>|<data>' where data is a base64 chunk (default 800 bytes) of the deflate-compressed ADF JSON.
- The overlay has fps and chunk-size controls that take effect live while cycling.
- On a non-Confluence page, an unrecognized URL, or a failed API fetch, the overlay shows a clear error message instead of a QR.
- An extension test page proves the browser-side round trip: ADF JSON compressed, chunked, and encoded, then decoded and reassembled back to byte-identical JSON.
- ADFReader's fixture list includes a Scan entry that opens the scanner screen.
- The chunk collector accepts frames in any order, ignores duplicates, and resets when a frame with a different docId arrives.
- While scanning, a segmented per-chunk progress bar and an 'n of total' count update as new chunks land, with a haptic tick on each new chunk.
- If no new chunk arrives for a few seconds mid-scan, a 'Keep the code in frame' hint appears.
- On collecting all chunks: success haptic and a brief checkmark, then automatic navigation to the rendered document via the existing ADF pipeline.
- If decompression or parsing fails after assembly, an error message with a 'Scan again' button appears and resets scanner state cleanly.
- The scanner screen accepts pasted chunk payloads (dev path), driving the identical collection/assembly/render pipeline without a camera.
- Swift unit tests verify chunk parsing, out-of-order reassembly, and inflate round-trip against fixture documents.
- Shared protocol fixtures (the same chunk payload files) pass verification on both the extension side and the Swift side, proving the two implementations agree.
- ADFReader declares NSCameraUsageDescription and the camera scanner works on a physical device (verified opportunistically; not required for acceptance).
